[LeetCode](https://leetcode.com/) 对任何程序员来说都是一个惊人的资源。无论您是否需要练习 SQL 查询、动态编程算法或字符串操作，LeetCode 肯定会为您带来一些问题。

但是就像任何其他解决编程问题的网站一样，LeetCode 的用处取决于你对它的描述。像 [HackerRank](https://www.hackerrank.com/) 和 [AlgoExpert](https://www.algoexpert.io/product) (几乎是 LeetCode 的精确克隆)这样的网站有论坛或讨论页面，程序员可以在那里分享和讨论他们对各种编码问题的解决方案。虽然这些论坛本身对于不确定如何解决特定问题或如何进一步优化代码的程序员来说是很好的资源，但它们也可能成为不必要的依靠。

# 学习错觉

我觉得很难给“学习”下定义。我可以非常抽象地说，“自身的进步”是定义学习的东西。虽然这个定义是正确的，但它不是非常具体，或特别有意义。也许我们可以把学习的范围缩小到一个特定的主题，比如数学，那么定义学习就会变得简单一些。

所以问问你自己:“学数学意味着什么？”现在我们可以更具体地定义学习了。我们可以这样回答:“看着你以前从未解决过的数学问题，形成新的神经连接，这样你就可以解决这样的问题。”但是我们如何从没有能力解决一个问题到有能力解决它呢？

想象一下，你在高中上数学课，你被布置了课本上的一些问题作为家庭作业。教科书后面有你所有问题的解决方案(像任何好的教科书一样)。你知道参考所有问题的解决方案是个坏主意，因为你实际上是在抄下答案。你肯定学不到任何东西。

所以你去做你的作业。前几个问题很容易，但是到了作业的中间，你发现自己被难住了。你尝试了跳过一个困难的问题，稍后再回到这个问题，看看你解决问题的大脑中是否有什么变化的可靠方法，但即使这样也不起作用。你告诉自己，“我只是快速浏览一下解决方案——只是想知道我应该做什么。”你看一看课本，对你看到的答案并不感到惊讶。“哦耶！我早就想到解决办法了！有道理……”

![](img/dfb7abb28dc4d68ab2a184dd1dc84e95.png)

在生死关头，千万不要和西西里人对抗！

你成了一个经典错误的受害者！当试图解决一个问题并给出一个解决方案时，我们通常会将这个解决方案(几乎总是有新奇的元素)与我们自己的解决问题的策略进行比较，并合理解释如果给我们更多的时间，我们也会提出同样的解决方案。

芭芭拉·奥克利在《T2:数字思维》中经常讨论这个概念。*是一本教读者如何更有效地学习的书，尤其是在 STEM 主题方面。在第四章中，这本书区分了真正的学习和简单地“理解”一个概念，后者是当我们错误地说服自己，我们会想出与答案相同的解决方案时发生的事情。换句话说，我们欺骗自己相信我们学到了一些东西，而事实上，我们只是简单地理解了一个特定的解决方案对应于一个特定的问题。*

*真正的学习发生在问题和它的正确解决方案之间形成深层联系的时候，而这种联系只会因为你如何“真诚地”解决问题而加深。*

*我想说清楚的是，问题和它的解决方案之间的联系存在于一个光谱上，而不是“你学会了如何解决问题或者你没有。”如果你能够解决问题的一半，然后参考一个解决方案来完成它，这仍然比从一开始就抄下一个答案要好。*

# *我应该使用 LeetCode 论坛吗？*

*是的，但是要非常精确。我在这里不是要告诉你 LeetCode 论坛是某种禁果，无论如何都不能碰。相反，我在这里给你一些建议，告诉你如何利用 LeetCode 论坛来增加你的学习优势(而不是劣势)。*

*你可以避免做的最重要的事情是在阅读完问题描述后立即查看 LeetCode 论坛。即使是简单的问题也需要一些时间来构思解决方案，所以不要期望简单的问题在阅读后的 30 秒内就能解决。不耐烦只会导致寻找解决方案，陷入前面提到的“学习错觉”*

*那么什么时候看 LeetCode 论坛合适呢？老实说，这取决于你。如果你认为你忘记了某种类型的语法(而不是不知道解决问题的算法)，那么我会建议你专门搜索你用来解决问题的语言的语法。如果广泛的搜索没有产生结果，那么可能是时候看一眼 LeetCode 答案，看看它是否包含您所缺少的语法。*

*我使用“浏览”这个词是因为如果你还没有提交一个被接受的答案，那么深入研究一个解决方案最终对你没有多大好处。过于仔细地看 LeetCode 论坛的答案会产生和你把数学课本后面的答案复制到笔记本上一样的效果。*

*这并不意味着当别人的 Python 代码出现在你的屏幕上时，你应该转移视线；只是不要专注于问题是如何解决的，或者这个人具体给变量/对象起了什么名字，等等。*

*就我个人而言，我倾向于避免 LeetCode 论坛，直到我可以提交一个可接受的解决方案。我观察其他人如何解决问题的主要原因是为了看看我可以改进的地方——我可以改进我的大 O 吗？我是不是做了一些不必要的操作？*

*当我最终在 LeetCode 论坛上看到我的代码如何变得更高效或更优雅时，我不会花很长时间看其他解决方案。我通常只看顶部的解决方案，在大约两三秒钟内浏览代码——足够我理解他们使用了什么数据结构或技术，而我在编写我的解决方案时已经忘记了。*

*如果我花超过 5 秒钟看别人的代码，我会很自然地开始记忆他们的解决方案(不是一个好主意)，如果我给自己 30 秒钟，我几乎肯定会将所有代码存储到我的短期记忆中，以便记录到我自己的代码编辑器中(一个非常糟糕的主意)。*
